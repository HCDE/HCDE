# HCDE Doom Connector Launch Contract

Date: 2026-05-11

This document is the stage 2 launch contract between Doom Connector and HCDE.
It names the current HCDE protocol boundary without claiming the deeper Odamex
server-authoritative replication port is complete.

## Dedicated Server Process

Doom Connector should launch a separate server process:

```powershell
hcdeserv -server <player-slots> -iwad <iwad-path> -port <udp-port> +map <map-name>
```

Example:

```powershell
hcdeserv -server 4 -iwad C:\Games\doom2.wad -port 10666 +map MAP01
```

Notes:

- `<player-slots>` is the number of visible player slots.
- HCDE internally reserves one hidden network slot for the dedicated server
  while the current UZDoom-era handshake remains in place.
- `+map` is preferred over `+changemap` for initial dedicated startup because
  the server must establish its initial map before clients join.
- Server discovery is opt-in:

```powershell
hcdeserv -server 4 -iwad C:\Games\doom2.wad -port 10666 -advertise -master hcde.servebeer.com:15000 +map MAP01
```

## Join Client Process

Doom Connector should launch each local HCDE player as a separate client:

```powershell
hcde -join <host:port> -dedicatedjoin -iwad <iwad-path> -file <mods...>
```

Example:

```powershell
hcde -join 127.0.0.1:10666 -dedicatedjoin -iwad C:\Games\doom2.wad -file C:\Mods\brutal.pk3
```

Notes:

- `-dedicatedjoin` hides the server/arbitrator slot from the in-game player
  list.
- `-netwaitsilent` is optional for launcher-controlled joins that should hide
  the old UZDoom room/session window.
- Doom Connector should not launch a second local HCDE player unless the user
  explicitly requested a second local client.

## Current Runtime Diagnostics

When HCDE enters a host/join path, it now prints `HCDE server runtime` lines
with:

- mode and executable type
- requested, visible, and internal slot counts
- reserved server authority slot state
- UDP port
- IWAD argument and selected IWAD
- map argument
- master advertising state
- old room UI suppression state

These diagnostics are meant to make bad launcher arguments obvious in normal
logs.

## Stage Boundary

This contract still rides on the current HCDE/UZDoom room handshake and tic
sync. The next networking stages must replace that with Odamex-style
connect/auth, service messages, snapshots, and server-owned gameplay authority.

## Stage 3 Update

Clients launched with `-dedicatedjoin` or `-netwaitsilent` now negotiate an
HCDE service-connect extension with the dedicated server. Doom Connector does
not need new command-line arguments for this stage. It should keep sending
`-dedicatedjoin` for dedicated joins and reserve `-netwaitsilent` for cases
where hiding the pregame room/status window is intentional:

```powershell
hcde -join <host:port> -dedicatedjoin -iwad <iwad-path> -file <mods...>
```

The server logs `HCDE service connect` when this path is active. If a dedicated
server logs `Legacy setup connect`, the launcher reached the server through the
old setup path and should be checked for missing join flags.

## Stage 4 Update

Once service connect is negotiated, dedicated joins now exchange pregame state
with HCDE service messages instead of the legacy room packet names. Doom
Connector still launches the same two-process shape:

```powershell
hcdeserv -server <player-slots> -iwad <iwad-path> -port <udp-port> -noadvertise +map <map-name>
hcde -join <host:port> -dedicatedjoin -iwad <iwad-path> -file <mods...>
```

The client should log `Received HCDE service start` before leaving the pregame
loop. If it logs only legacy `Received GO`, the server and client did not enter
the stage 4 service-message path.

## Stage 5 Update

Dedicated service joins now wait for explicit server-authored console-player
and map-load messages before starting. A healthy join log should include:

- `Waiting for server assignment`
- `Received HCDE console player <slot>`
- `Received HCDE map load`
- `Received HCDE server info`
- `Received HCDE service start`

Doom Connector does not need new command-line arguments for this stage.

## Stage 6 Update

HCDE pregame service messages now carry a service sequence and peer ACK, with
duplicate/replay protection on both sides of the dedicated join. Doom Connector
does not need new command-line arguments for this stage.

A healthy join should still use the same launch shape:

```powershell
hcdeserv -server <player-slots> -iwad <iwad-path> -port <udp-port> -noadvertise +map <map-name>
hcde -join <host:port> -dedicatedjoin -iwad <iwad-path> -file <mods...>
```

The service ACK fields are internal to HCDE and should not be generated by Doom
Connector.

## Stage 7 Update

HCDE now retains required pregame service messages and retransmits the oldest
unacknowledged message with its original sequence number until the peer ACKs it.
Doom Connector does not need new command-line arguments for this stage.

Keep launching the dedicated pair as:

```powershell
hcdeserv -server <player-slots> -iwad <iwad-path> -port <udp-port> -noadvertise +map <map-name>
hcde -join <host:port> -dedicatedjoin -iwad <iwad-path> -file <mods...>
```

Launcher-side behavior is unchanged: start exactly one `hcdeserv` for the room
and exactly one `hcde` per local player requested by the user.

## Stage 8 Update

HCDE service clients now acknowledge `HPS_START_GAME` before the dedicated
server leaves the pregame service loop. Doom Connector does not need new
command-line arguments for this stage.

A healthy dedicated server log should include:

- `Starting dedicated game`
- `Go`
- `Client <slot> acknowledged HCDE service start`
- `Total players: <count>`

## Stage 9 Update

Retained HCDE setup messages now time out after 15 seconds if the peer never
ACKs them. Doom Connector does not need new command-line arguments for this
stage.

If a join stalls, the dedicated server log now identifies the exact retained
service that timed out:

- `Client <slot> timed out during <phase> on HCDE service <service>`
- `Client <slot> disconnected from server`

The join client receives `The server timed out during HCDE setup` instead of
waiting forever.

## Stage 10 Update

Negotiated HCDE service peers now keep a separate `HLIV` live-game packet lane
after pregame. Doom Connector does not need new command-line arguments for this
stage.

This stage is diagnostic transport only: HCDE sends a small live control frame
once per second between the dedicated server and each negotiated client. Client
commands and server snapshots still need later stages before the inherited peer
tic protocol can be removed.

## Stage 11 Update

Negotiated HCDE service clients now send gameplay command packets to the host
inside `HLIV` client-command frames. Doom Connector does not need new
command-line arguments for this stage.

A healthy dedicated join log should include:

- `NetSession:: HCDE live client commands active with client 0`
- `NetServer:: HCDE live client commands active with client <slot>`

This stage still uses the inherited tic-command parser after the host unwraps
the live payload. Server snapshots and fully server-owned gameplay authority are
still later networking stages.

## Stage 12 Update

Negotiated HCDE dedicated hosts now send gameplay traffic to clients inside
`HLIV` server-snapshot frames. Doom Connector does not need new command-line
arguments for this stage.

A healthy dedicated join log should now include all four live-lane messages:

- `NetSession:: HCDE live client commands active with client 0`
- `NetServer:: HCDE live client commands active with client <slot>`
- `NetServer:: HCDE live server snapshots active with client <slot>`
- `NetSession:: HCDE live server snapshots active with client 0`

This stage still unwraps the server-snapshot payload into the inherited
tic-command parser. Explicit server-authored world-state deltas and fully
server-owned gameplay authority are still later networking stages.

## Stage 13 Update

Live `HLIV` client-command and server-snapshot frames now carry an HCDE gameplay
envelope before the temporary legacy payload. Doom Connector does not need new
command-line arguments for this stage.

The envelope records:

- gameplay-envelope version
- payload kind
- room id
- sender game tic

HCDE validates and strips that envelope before the inherited tic-command parser
sees the legacy payload. Explicit input messages, world-state deltas, and fully
server-owned gameplay authority are still later networking stages.

## Stage 14 Update

Client-to-server live movement now uses an explicit `HCIN` client-input payload
inside the Stage 13 `HGPL` gameplay envelope. Doom Connector does not need new
command-line arguments for this stage.

A healthy dedicated join log should now include:

- `NetSession:: HCDE live client inputs active with client 0`
- `NetServer:: HCDE live client inputs active with client <slot>`
- `NetServer:: HCDE live server snapshots active with client <slot>`
- `NetSession:: HCDE live server snapshots active with client 0`

The host validates the `HCIN` input metadata before rebuilding the temporary
inherited tic packet for the current parser. Stage 15 gives server snapshots
their own `HCSN` payload before explicit state deltas land.

## Stage 15 Update

Host-to-client live snapshots now use an explicit `HCSN` server-snapshot payload
inside the Stage 13 `HGPL` gameplay envelope. Doom Connector does not need new
command-line arguments for this stage.

A healthy dedicated join log should now include:

- `NetSession:: HCDE live client inputs active with client 0`
- `NetServer:: HCDE live client inputs active with client <slot>`
- `NetServer:: HCDE live server snapshots active with client <slot>`
- `NetSession:: HCDE live server snapshots active with client 0`

The client validates `HCSN` snapshot metadata before rebuilding the temporary
inherited snapshot packet for the current parser. Fully server-authored
world-state deltas and dedicated server authority are still later networking
stages.

## Stage 16 Update

Client-to-host live inputs now use explicit `HCIR` player, consistency, and
command records inside the Stage 14 `HCIN` payload. Doom Connector does not need
new command-line arguments for this stage.

A healthy dedicated join log should still include:

- `NetSession:: HCDE live client inputs active with client 0`
- `NetServer:: HCDE live client inputs active with client <slot>`
- `NetServer:: HCDE live server snapshots active with client <slot>`
- `NetSession:: HCDE live server snapshots active with client 0`

The host validates `HCIR` record counts, offsets, player ids, and command record
lengths before rebuilding the temporary inherited tic packet. Stage 18 and
Stage 19 later replace the client-input command and event byte edges.

## Stage 17 Update

Host-to-client live snapshots now use explicit `HCSR` player, consistency, and
command records inside the Stage 15 `HCSN` payload. Doom Connector does not need
new command-line arguments for this stage.

A healthy dedicated join log should still include:

- `NetSession:: HCDE live client inputs active with client 0`
- `NetServer:: HCDE live client inputs active with client <slot>`
- `NetServer:: HCDE live server snapshots active with client <slot>`
- `NetSession:: HCDE live server snapshots active with client 0`

The client validates `HCSR` record counts, player ids, latencies, offsets, and
command record lengths before rebuilding the temporary inherited snapshot
packet. Stage 21 later replaces those inherited snapshot command byte blobs
with HCDE event records and explicit user command fields.

## Stage 18 Update

Client-to-host live inputs now send explicit user command fields inside `HCIR`
command records. Doom Connector does not need new command-line arguments for
this stage.

The `HCIR` command record now separates queued net events from movement input:

- command tic offset
- bounded event byte count and event bytes
- `buttons`
- `pitch`
- `yaw`
- `roll`
- `forwardmove`
- `sidemove`
- `upmove`

The host validates the record shape and rebuilds the temporary inherited user
command only after the wire payload has been decoded. Queued net events are
still byte-streamed inside a length-bounded section until they get explicit HCDE
event records.

## Stage 19 Update

Client-to-host live inputs now send explicit event records inside `HCIR`
command records. Doom Connector does not need new command-line arguments for
this stage.

The `HCIR` command record now separates events into:

- event count
- event type
- event payload byte count
- event payload bytes

The host validates event record boundaries and event types before rebuilding the
temporary inherited event stream for the current parser.

## Stage 20 Update

Client-to-host live input event payloads now use HCDE-owned schemas inside
`HCIR` event records. Doom Connector does not need new command-line arguments
for this stage.

The `HCIR` event record still starts with:

- event type
- event payload byte count

The payload behind that count is no longer an inherited raw byte copy. HCDE now
validates each supported event type with its own shape, including
length-prefixed strings, explicit cvar updates, explicit weapon-slot indices,
fixed numeric fields, and bounded script/event argument blocks. The host only
rebuilds the temporary inherited event stream after this schema validation
passes.

## Stage 21 Update

Host-to-client live server snapshots now send explicit `HCSR` command records
instead of inherited command/event byte blobs. Doom Connector does not need new
command-line arguments for this stage.

The `HCSR` command record now stores:

- command tic offset
- event record count
- zero or more Stage 20 event records
- explicit user command fields

The client validates the record counts, offsets, event schemas, and fixed
command fields before rebuilding the temporary inherited snapshot packet for the
current gameplay parser.

## Stage 22 Update

Dedicated HCDE hosts now enforce client input ownership before rebuilding the
temporary inherited input packet. A negotiated dedicated client may submit either
an empty input heartbeat or one `HCIR` player record for its own assigned slot.
Packets that try to carry another player's input, or multiple player records,
are rejected at the live-input boundary.

Doom Connector does not need new command-line arguments for this stage. Keep
using the dedicated launch pair:

```powershell
hcdeserv -server <visible-player-count> -port <port> -iwad <iwad-path> -file <mods...> +map <map>
hcde -join <host:port> -dedicatedjoin -iwad <iwad-path> -file <mods...>
```

## Stage 23 Update

Host-to-client live server snapshots now include an `HCDW` server-authored
world-delta section after the `HCSR` command records. The first delta stream
carries authoritative player pose records for the players represented in the
snapshot. Clients validate this section before the temporary inherited snapshot
packet is rebuilt.

Doom Connector does not need new command-line arguments for this stage.

## Stage 24 Update

Clients now consume validated `HCDW` player pose records for baseline-repair
telemetry. Remote player baseline drift is detected from the server-authored
snapshot, but direct correction is deferred while inherited lockstep
consistency still owns playsim state.

Doom Connector does not need new command-line arguments for this stage.

## Stage 25 Update

Clients now validate the local predicted player against `HCDW` player pose
records and report reconciliation drift. The actual playsim mutation path is
guarded off until live replication replaces inherited lockstep consistency.

Doom Connector does not need new command-line arguments for this stage.

## Stage 26 Update

Dedicated server sessions now hide the reserved server slot behind explicit
slot-mapping helpers. Public player counts, server-query snapshots, pregame
progress, status logging, and prediction gating now use the slotless boundary
instead of exposing the inherited slot-zero arbitrator as a playable client.

Doom Connector does not need new command-line arguments for this stage.

## Stage 27 Update

HCDE live packet routing now uses explicit authority-route helpers instead of
direct `Net_Arbitrator` checks in the live `HLIV` send/receive layer. The
current internal authority slot is unchanged, but the routing decision is now
centralized so the next stages can replace player-backed server state without
rewriting every live-packet gate again.

Doom Connector does not need new command-line arguments for this stage.

## Stage 28 Update

Dedicated HCDE server sessions now keep local settings authority in HCDE server
session state instead of relying on the reserved internal server slot being a
playable settings controller. Console commands, server CVAR changes, save/pause
checks, and bot/admin commands now ask the same authority helper.

Doom Connector does not need new command-line arguments for this stage.

## Stage 29 Update

HCDE now has a service-authority identity boundary for new service/live netcode.
Live routing, reserved-slot mapping, and dedicated join assignment validation
now ask HCDE authority helpers instead of directly comparing playable clients to
`Net_Arbitrator`.

Doom Connector does not need new command-line arguments for this stage.

## Stage 30 Update

The inherited runtime packet receive/start boundary now also asks HCDE
service-authority helpers. Packet sizing, disconnect forwarding, level-start
coordination, host-authored receive fields, and authority heartbeat gating no
longer open-code direct `Net_Arbitrator` comparisons in that slice.

Doom Connector does not need new command-line arguments for this stage.

## Stage 31 Update

Inherited tic balancing and packet construction now use the same HCDE
service-authority helpers as the receive/start boundary. Host-side packet
sequence selection, consistency emission, stability pacing, and latency fields
no longer open-code direct `Net_Arbitrator` comparisons in that slice.

Doom Connector does not need new command-line arguments for this stage.

## Stage 32 Update

Local quit and host-migration packet emission now uses HCDE service-authority
helpers. Dedicated server sessions skip the reserved server slot when choosing a
replacement authority, while the wire packet shape remains compatible with the
current inherited tic protocol.

Doom Connector does not need new command-line arguments for this stage.

## Stage 33 Update

Read-only status and notification paths now use HCDE service-authority helpers.
The network stat view, cutscene host-only readiness, FOV/settings-controller
notifications, and ping visibility no longer describe or filter the authority
through direct inherited arbitrator checks.

Doom Connector does not need new command-line arguments for this stage.
